# 2026-05-13 — Sprint 1 (engine v9) + Sprint 2 (AIS service)

## What we worked on

Closed out the "pillar 1 — plan race from forecasted weather at start"
correctness gaps in a single push, plus shipped the standalone AIS
service that Stream 3 of the plan called for.

Pillar-1 gaps that landed:

* **Multi-leg routing.** Engine now threads wall-clock time across an
  arbitrary number of marks. Previously took only `start = marks[0]` and
  `finish = marks[-1]`, ignoring every intermediate mark — a real
  correctness bug for any MORF / windward-leeward course.
* **Rounding side (port / starboard) per intermediate mark.**
  Enforced with a half-plane constraint near each rounded mark plus a
  200 m seed offset for the next leg.
* **Polar extensions** — `boat_speed(twa, tws, *, hs_m, density_factor,
  margin)`. All three default to no-op so legacy callers (existing
  tests) keep passing unchanged.
* **Surface-current support** — engine accepts a `currents=` sampler
  that vector-adds to projected boat position. Not wired in the router
  yet (Stream 2: GLERL GLCFS ingest worker), but the engine API is in
  place.
* **Heavy-weather cutoff** — `max_tws_kt` param rejects expansions from
  any frontier point where forecast TWS exceeds the cutoff.
* **Polar margin (gust de-rating)** — global multiplier in [0, 1]. The
  router defaults to 0.97 ⇒ baked-in 3% conservatism vs polar
  idealization. The "no defer gusts" item from the plan; handled via
  polar margin rather than per-sample gust ingest, per the agreed
  approach.
* **Air density factor** — exposed as `density_factor` on polar. Until
  temp/pressure ingest lands (Stream 2 extension of weather_ingest), the
  router accepts it from the client and threads it through.
* **Migration 0006** — backfills `rounding: null` on every existing
  marks JSONB entry so the frontend can rely on the key being present.

Sprint 2 (AIS) — service layer + worker + read endpoint, display-only:

* **`app/services/ais.py`** — `AisPosition` dataclass, Redis
  read/write helpers, AISStream.io message parsing (PositionReport +
  ShipStaticData identity merge), `consume_messages` async iterator,
  `run_subscription` reconnect loop.
* **`app/routers/ais.py`** — `GET /api/ais?min_lat=&max_lat=&min_lon=&max_lon=`
  reads cached positions, auth-gated, capped at 500 per response.
* **`workers/ais_subscribe.py`** — standalone CLI entrypoint. Defaults
  to every venue bbox from `app/regions.py` so it's zero-config in
  production. Documents the dedicated-Cloud-Run-service deployment
  pattern (min_instances=1, max_instances=1).

## Decisions and rationale

* **Polar margin instead of gust ingest.** Cheaper than ingesting,
  parsing, and per-sample handling gust fields. A 3% global de-rating
  captures the gust effect at the cost-of-fidelity boundary that v1
  cares about. Real per-sample gust handling can be revisited in v1.5
  if route quality complaints surface.
* **Multi-leg as a separate function, not a refactor of single-leg.**
  Kept `compute_isochrone_route` as the single-leg primitive so the
  existing 4 test files exercising it (test_isochrone_engine,
  test_isochrone_time_threading, …) pass unchanged. The multileg
  function is a driver that calls the single-leg primitive once per
  leg.
* **Rounding via half-plane constraint + 200 m seed offset.** The
  rounding filter rejects candidates on the wrong side of the
  mark→next-mark bearing within 3 nm of the mark. The 200 m seed
  offset displaces the next leg's start position to the correct side
  so the engine's heading sweep doesn't immediately self-reject. Two
  layers because either alone leaves a degenerate case.
* **AIS as a standalone worker, not a FastAPI background task.** Cloud
  Run autoscales API instances — running the WS subscription inside
  the main app would N-multiply traffic and burn AISStream rate
  limits. The worker is a separate Cloud Run service with
  min_instances=1 / max_instances=1, behind the same VPC connector so
  it can write to the shared Memorystore.
* **AIS Redis schema: per-MMSI key + Python-side bbox filter.** At
  <2 000 vessels in any race area, a SCAN + filter is plenty fast.
  Avoids the operational complexity of GEOADD when we don't need it
  yet.
* **AIS engine integration parked.** Display-only is the v1
  commitment. Hooking AIS into the navigability predicate (avoidance
  corridors) is v2 per the plan — the AISPosition shape carries
  enough info (cog, sog) to drive a moving-obstacle predicate later
  without re-fetching.
* **AISStream.io confirmed cheaper than Datalastic.** Datalastic
  starts at €199/mo (Starter 20k req). Polling at 30s for one active
  race = 86k req/month → forces the €679/mo unlimited tier. AISStream
  is free in beta with no published rate cap. Acceptable since v1 AIS
  is non-safety-critical.
* **Currents not yet wired in the router.** Engine accepts the
  argument, router passes None. GLCFS ingest worker is Stream 2.
  Leaving the wiring in place so dropping currents in is a
  one-line change to the router.

## Technical debt flagged

* **`scripts/isochrone.py`** still imports the single-leg API. It's a
  standalone CLI, not part of the test suite or deploy path, so it
  works as-is, but if someone runs it on a multi-mark course expecting
  v9 behavior they'll get start-to-finish only. Update or delete in a
  future cleanup pass.
* **AIS bbox query is a SCAN.** Fine at v1 volumes; revisit if vessel
  count grows past 2 000 per region — switch to GEOADD/GEOSEARCH.
* **Polar wave-derating model is hand-coded.** Linear penalty with
  hand-picked constants (5%/m up, capped at 20%; +1.25%/m down,
  capped at +5%). Replace with per-class measured derating tables once
  we calibrate against on-water data.
* **No CI smoke check that pytest passes.** The bash mount lag
  blocks in-sandbox runs; if this becomes routine the project could
  use a "trigger CI on every code change" hook so Cloud Build catches
  failures instead of waiting on Windows-side runs.
* **Background recompute uses hardcoded derating defaults.** If we
  ever persist user derating preferences (e.g. "this skipper always
  routes at margin 0.95") the recompute worker should read those
  back, otherwise its "better route" alert may differ from what the
  user would have computed manually.
