# Race Tracking Improvements — Multi-Session Plan

**Date:** 2026-05-14
**Trigger:** First real-world live tracking test (Cook County Sailing, 2026-05-13). Five observations surfaced.
**Strategy:** Each session below is sized to fit a single context window. Do them in order — later sessions depend on earlier ones.

---

## Observations from the 2026-05-13 race

1. **Phone sensor consistency** — phone moving around (pocket, hand) makes telemetry inconsistent.
2. **Tracking starts manually** — should auto-start ~5 min before race `start_at`.
3. **Tracking stops when screen locks** — browser `watchPosition` is paused when the tab is hidden. Pure-web fix is not viable.
4. **Tracking has to be stopped manually** — should auto-stop after the final mark is passed (+5 min buffer), but only after the second-to-last mark has been hit (so beer-can races whose start ≈ finish don't stop on the first lap).
5. **No post-race stats view** — need a race-specific stats screen (no health metrics) similar to the Garmin Connect screenshot.

## Decisions

| # | Decision | Rationale |
|---|---|---|
| 1 | **Drop "calibration" as code work.** Treat as user guidance only — instruct racers to mount the phone in a fixed location (cockpit clip, RAM mount, gimbal). Add a one-line tip on the race-start screen. | No sensor calibration is actually needed; the user's concern resolves to consistent physical placement. Cheap to address without code. |
| 2 | **Adopt Capacitor** as the native shell. | Background geolocation is impossible in a pure PWA. Capacitor wraps the existing React/Vite build with minimal code change and unlocks iOS/Android distribution. The app needs to be installable on a phone for real race use; this is overdue. |
| 3 | Each numbered improvement gets its own session. | Capacitor work alone is multi-day; bundling with feature work risks an incomplete state in any one session. |

## Session order

1. **Session A — Capacitor wrapper + background geolocation** (Obs. #3) — biggest, blocks nothing else technically but unblocks real-world racing.
2. **Session B — Auto-start 5 min before race** (Obs. #2) — small, independent. Run in parallel with A if Session A spans days.
3. **Session C — Mark-rounding detector + auto-stop** (Obs. #4) — net-new service; prerequisite for D.
4. **Session D — Post-race stats view** (Obs. #5) — depends on C's leg-split timestamps for the leg-by-leg breakdown.

## Session A — Capacitor wrapper + background geolocation

**Goal:** Ship a Capacitor-wrapped build of the existing React app with true background GPS tracking that survives screen lock.

**Scope**
- Add `@capacitor/core`, `@capacitor/cli`, `@capacitor/ios`, `@capacitor/android` to `frontend/package.json`.
- `npx cap init` with appId `com.sailline.app`, name `SailLine`.
- Add `@capacitor/geolocation` and a community background-geolocation plugin (likely `@capacitor-community/background-geolocation` — confirm at session start; pick the most actively maintained option).
- Create a `geolocation.js` adapter in `frontend/src/lib/` that abstracts "browser `watchPosition`" vs. "Capacitor background watcher" behind one interface. `useTrackRecorder` calls the adapter — no platform branching inside the hook.
- iOS: configure `NSLocationAlwaysAndWhenInUseUsageDescription`, `UIBackgroundModes: location` in `Info.plist`.
- Android: request `ACCESS_BACKGROUND_LOCATION`, declare a foreground service in `AndroidManifest.xml`, show a persistent notification while recording (Android requirement post-API 29).
- Build: `frontend/capacitor.config.ts` pointing webDir to `dist`. CI continues to build the web app the same way; native builds are local-only for now (Cloud Build pipelines unchanged).
- Smoke test on a real device — must record a 15-minute track with the screen locked.

**Files expected to change / be added**
- `frontend/package.json` (deps)
- `frontend/capacitor.config.ts` (new)
- `frontend/src/lib/geolocation.js` (new — platform-adaptive watcher)
- `frontend/src/hooks/useTrackRecorder.js` (swap direct `navigator.geolocation` calls for the adapter)
- `frontend/ios/` and `frontend/android/` (generated by `cap add`)
- `sailline -docs/2026-05-14-session-summary-capacitor.md` (session summary)

**Acceptance**
- Web build still works unchanged in the browser.
- On a physical iPhone or Android device: record a track, lock the screen for 10+ minutes, verify points captured continuously with no gap.
- The persistent foreground-service notification (Android) shows correctly and dismisses when recording stops.

**Risks / open items**
- Background-geolocation plugin choice. Some plugins are abandoned or have license restrictions. First task in Session A is a 30-min triage of plugin options.
- App store distribution (TestFlight, Play Internal Testing) is out of scope for this session — local-device install only.
- Native app review and ongoing iOS dev account cost ($99/yr Apple Developer). Surface as cost decision before opening Xcode.
- **Tech debt flag:** Capacitor introduces a second build target. CI doesn't build native; if web-only tests pass but a native-only path breaks, we won't catch it. Acceptable for v1 since the only platform-specific code is the geolocation adapter, but worth revisiting.

## Session B — Auto-start 5 minutes before race

**Goal:** Recorder begins capturing automatically at `race.start_at - 5 min` when the race is open in the app.

**Scope**
- New `useAutoStartRecorder(race, recorder)` hook (or a `useEffect` inside `RaceEditor.jsx` / wherever recording is owned) that watches the clock against `race.start_at`.
- Behaviour: if `now >= start_at - 5min` and `recorder.recording === false`, call `recorder.start()`. Idempotent — if the user manually started earlier, do nothing.
- Show a small "Auto-recording armed at 7:55 PM" banner so the user knows what's about to happen.
- Respect a per-race opt-out (toggle in race settings: `auto_start_enabled`, default true).

**Files expected to change**
- `frontend/src/hooks/useAutoStartRecorder.js` (new)
- `frontend/src/RaceEditor.jsx` (wire the hook in)
- `frontend/src/components/RouteControls.jsx` or equivalent (toggle UI)
- `backend/migrations/000X_auto_start_flag.py` (additive column on `race_sessions`)
- `backend/app/routers/races.py` (expose flag in race read/write models)
- `frontend/src/__tests__/useAutoStartRecorder.test.js` (new — vitest, fake timers)

**Acceptance**
- With `start_at` set 6 minutes in the future, `recorder.start()` is called once when 5 minutes remain.
- Toggling `auto_start_enabled = false` prevents the call.
- Vitest passes locally on Windows. (Sandbox can't run vitest — remembered.)

**Risks / open items**
- Race-start time may slip after the app is loaded. Hook must re-evaluate on `start_at` change, not just on mount.
- Tab not open → no auto-start fires. Acceptable; not solving for "armed but app closed" in this session.

## Session C — Mark-rounding detector + auto-stop

**Goal:** Detect when the boat rounds each mark in sequence, and stop recording automatically 5 minutes after the final mark has been passed — but only once the second-to-last mark has also been passed (gates against beer-can start≈finish layouts).

**Scope**
- New service `backend/app/services/mark_rounding.py`:
  - `MarkRoundingDetector(course_marks, radius_m=50)` consumes track points in order and emits `MarkRounded(mark_index, timestamp)` events. Rounding = entered the radius AND exited it (not just a fly-by).
  - Optionally a bearing-change check for stricter "the boat actually turned the corner" detection — start simple (radius-only) and revisit only if false positives surface.
- Surface mark-pass state on the frontend, either via:
  - Server-side: extend `POST /api/races/{id}/track` response to include `mark_passes: [{mark_index, ts}]` and last-known `current_leg`, OR
  - Client-side: run the same detector in `frontend/src/lib/markRounding.js` against the in-memory `points` buffer (faster feedback, no extra round trips).
- Decision: client-side mirror for live UX, server-side authoritative pass for stats. Mirror pattern matches `regions.py` ↔ `regions.js`.
- Auto-stop logic in `useTrackRecorder` (or a sibling hook):
  - Track which mark indices have been rounded.
  - When `(course.length - 2)` has been rounded AND `(course.length - 1)` has been rounded AND `now >= last_rounding_ts + 5min`, call `recorder.stop()`.
  - Never stop before second-to-last is rounded — handles beer cans.

**Files expected to change / be added**
- `backend/app/services/mark_rounding.py` (new)
- `backend/app/routers/tracks.py` (extend response model; minor)
- `backend/tests/test_mark_rounding.py` (new — pure-function unit tests)
- `frontend/src/lib/markRounding.js` (new — JS mirror)
- `frontend/src/hooks/useTrackRecorder.js` (auto-stop wiring) OR new `useAutoStopRecorder.js` (preferred — keep recorder pure)
- `frontend/src/__tests__/markRounding.test.js`

**Acceptance**
- Unit tests cover: straight pass, fly-by miss, multi-lap start/finish (W-L), partial course (DNF case — never stops).
- Manual test: replay yesterday's `.fit` track through the detector offline, confirm reasonable rounding times for the Cook County Sailing course.

**Risks / open items**
- Mark radius is the only knob. Big buoys + sloppy GPS may demand 75m or more. Start with 50m, make it configurable per race in a follow-up if needed.
- Out-of-order rounding (e.g., bailing on a leg) — detector should not accept a later mark before an earlier one is rounded. Tests should cover this.

## Session D — Post-race stats view

**Goal:** A `RaceStatsView` page reachable from the races list (and shown automatically when recording auto-stops) that displays race-specific stats, no health metrics.

**Scope**
- Mirror the useful columns from the Garmin screenshot, drop anything physiological:
  - Distance sailed (great-circle integrated over track points)
  - Elapsed time, moving time, time stopped (speed < 0.5 kt threshold)
  - Average SOG, max SOG, average moving SOG
  - Leg-by-leg breakdown using Session C's mark-pass timestamps: distance, elapsed, avg SOG per leg
  - Map of the track (existing MapView, read-only mode)
  - Speed-over-time chart (lightweight — keep dependencies small; reuse existing chart libs if any, else SVG sparkline)
- Backend: a `GET /api/races/{id}/stats` endpoint that computes the stats server-side from `track_points` + the race's mark sequence. Cache result in Redis keyed on `(race_id, track_point_count)` so re-loads are cheap.

**Files expected to change / be added**
- `backend/app/routers/races.py` (new `/stats` endpoint) or new `backend/app/routers/race_stats.py`
- `backend/app/services/race_stats.py` (new — pure function: track + marks → stats)
- `backend/tests/test_race_stats.py`
- `frontend/src/RaceStatsView.jsx` (new)
- `frontend/src/hooks/useRaceStats.js` (new)
- `frontend/src/RacesListView.jsx` (add "View Stats" entry for completed races)
- `frontend/src/__tests__/RaceStatsView.test.jsx`

**Acceptance**
- Stats endpoint returns a sane payload for the 2026-05-13 Cook County Sailing test track.
- Frontend renders distance / time / avg+max SOG that matches Garmin within ~2% (small drift is OK — different sample rates).
- Leg splits correspond to mark-pass timestamps from Session C.

**Risks / open items**
- Existing `track_points` may have gaps from screen-lock incidents. Stats math must be tolerant of long sample gaps (don't integrate distance across an obvious teleport).
- Defining "moving" vs. "stopped" threshold is subjective. Default to 0.5 kt, configurable later if needed.
